# OTP Backend Service — Checkpoint

## Дата
2026-04-23

## Текущее состояние проекта
Проект приведен к рабочему и структурированному состоянию после серии рефакторингов и полного ручного регресса.

## Что реализовано

### 1. Базовая архитектура
Приложение построено по слоям:
- API / controllers
- service
- repository / DAO

Используется:
- Java 17
- Maven
- PostgreSQL 17
- JDBC
- `com.sun.net.httpserver` для HTTP API

### 2. Пользователи и роли
Реализовано:
- регистрация пользователя
- логин пользователя
- logout
- токенная аутентификация
- разграничение ролей USER / ADMIN
- запрет доступа обычного пользователя к admin API

### 3. OTP-функционал
Реализовано:
- генерация OTP-кода
- валидация OTP-кода
- статусы OTP:
  - ACTIVE
  - EXPIRED
  - USED
- настройка длины кода и TTL через admin API
- планировщик, переводящий истекшие коды в EXPIRED

### 4. Каналы доставки OTP
Реализованы и протестированы:
- FILE
- EMAIL
- TELEGRAM
- SMS через SMPP-эмулятор

### 5. Admin-модуль
Выполнен вынос административного API в отдельный модуль `admin`:
- `AdminUserController`
- `AdminUserService`
- `AdminOtpConfigController`
- `AdminOtpConfigService`

После рефакторинга:
- `user` отвечает только за пользовательскую часть
- `otp` отвечает за OTP-домен
- `admin` отвечает за административное API

### 6. Улучшения бизнес-логики OTP
Добавлено правило:
- для одной операции не допускается повторная генерация нового OTP, если предыдущий ACTIVE-код еще не истек
- вместо повторной генерации API возвращает понятное сообщение:
  - сколько секунд осталось ждать до повторной генерации

### 7. Полировка запуска приложения
Добавлен отдельный `StartupPrinter`:
- красивый startup banner
- шаги запуска
- summary по URL, scheduler, channels, endpoints

## Что было проверено вручную

### Auth
- login
- logout
- проверка невалидности токена после logout

### User API
- `GET /users/me`

### Admin API
- `GET /admin/users`
- `DELETE /admin/users/{id}`
- `GET /admin/otp-config`
- `PUT /admin/otp-config`

### OTP
- `POST /otp/generate`
- `POST /otp/validate`
- успешная валидация свежего кода
- повторная валидация использованного кода -> `valid=false`
- неверный код -> `valid=false`

### Каналы
- FILE — успешно
- EMAIL — успешно
- TELEGRAM — успешно
- SMS / SMPP — успешно

### Негативные сценарии
- обычный пользователь не имеет доступа к admin API
- EMAIL без email -> понятная ошибка
- неверный канал -> понятная ошибка
- повторная генерация активного OTP по той же операции -> понятная ошибка с оставшимся временем
