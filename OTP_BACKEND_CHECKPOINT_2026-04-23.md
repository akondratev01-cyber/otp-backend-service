# OTP Backend Service — Checkpoint

## Дата чекпоинта
2026-04-23

## Что реализовано и проверено

### 1. Аутентификация и пользователи
Реализованы и протестированы endpoint'ы:

- `POST /auth/register`
- `POST /auth/login`
- `POST /auth/logout`
- `GET /users/me`
- `GET /admin/users`
- `DELETE /admin/users/{id}`

Что подтверждено:
- регистрация пользователя работает;
- логин выдает токен;
- логаут инвалидирует токен;
- получение текущего пользователя по токену работает;
- админ может получать список пользователей;
- админ может удалять пользователя по `id`.

---

### 2. OTP core
Реализованы и протестированы:

- `POST /otp/generate`
- `POST /otp/validate`

Что подтверждено:
- OTP-код создается в БД;
- корректный код валидируется успешно;
- повторная валидация того же кода возвращает `false`;
- после успешной проверки код переводится в `USED`.

---

### 3. OTP configuration
Реализованы и протестированы admin endpoint'ы:

- `GET /admin/otp-config`
- `PUT /admin/otp-config`

Что подтверждено:
- текущая конфигурация OTP читается через API;
- конфигурация обновляется через API;
- поля `codeLength`, `ttlSeconds`, `updatedAt` корректно возвращаются в JSON;
- для сериализации `LocalDateTime` добавлен модуль Jackson `jackson-datatype-jsr310`.

---

### 4. Scheduler и EXPIRED
Реализован `OtpExpirationScheduler`.

Что подтверждено:
- новый код сначала имеет статус `ACTIVE`;
- после истечения TTL scheduler переводит код в `EXPIRED`;
- проверка сценария выполнена на сокращенном TTL через `otp_config`.

---

### 5. Канал FILE
Реализован `FileOtpNotificationService`.

Что подтверждено:
- при генерации OTP с каналом `FILE` создается файл:
  - `var/otp/otp-codes.log`
- в файл записываются:
  - `userId`
  - `operationId`
  - `code`

---

### 6. Канал EMAIL
Реализован `EmailOtpNotificationService`.

Что подтверждено:
- email обязателен для канала `EMAIL`;
- SMTP-настройки вынесены в локальный файл `application-local.properties`;
- `ApplicationProperties` доработан так, чтобы:
  1. сначала читать `application.properties`
  2. затем поверх него читать `application-local.properties`
- email-отправка успешно протестирована через Gmail SMTP;
- письмо реально приходит на почту;
- письмо переведено на HTML-формат и теперь красиво оформлено:
  - крупный заголовок `Your OTP Code`
  - выделенный блок с кодом
  - `Operation ID`
  - `User ID`

---

## Дополнительные технические решения

### Безопасность локальных секретов
Для SMTP использован отдельный локальный конфиг:

- `src/main/resources/application-local.properties`

План использования:
- реальные секреты хранятся только локально;
- основной `application.properties` может содержать безопасные заглушки;
- в `.gitignore` добавлена запись:
  - `src/main/resources/application-local.properties`

## Текущее состояние архитектуры
На текущем этапе admin-сценарии реализованы внутри доменных модулей:

- admin операции по пользователям — в `user`
- admin операции по OTP-конфигу — в `otp`

Папка `admin` сейчас фактически остается пустой заготовкой и пока не используется как отдельный модуль.

Решение на текущем этапе:
- **пока не рефакторить**
- при необходимости вернуться к структуре ближе к финалу проекта

---

## Что уже можно уверенно демонстрировать
Проект уже умеет:

- регистрировать пользователей;
- логинить пользователей и выдавать токены;
- выполнять логаут;
- получать текущего пользователя по токену;
- выполнять admin-операции над пользователями;
- генерировать OTP;
- проверять OTP;
- переводить OTP в `USED`;
- автоматически переводить просроченные OTP в `EXPIRED`;
- читать и обновлять OTP-конфиг через admin API;
- сохранять OTP в файл;
- отправлять OTP по email в красивом HTML-виде.
